# harness-explore：探索模式

## 触发条件

用户想改代码但**不确定改什么、改多大范围、用什么方案**时执行。

适用场景：
- 性能优化（不知道瓶颈在哪）
- 调试复杂问题（不确定根因）
- 重构决策（不确定影响范围）
- 技术选型（需要对比方案）

## 输入

- `project.yaml`（必须存在）
- 用户的模糊描述或问题
- `docs/meta/project-index.yaml`（如存在，优先读取以快速定位文件）

## 步骤

### 1. 明确探索目标

向用户提问：

> 你想探索什么问题？描述一下现象、困惑或者想优化的方向。不用给出方案，我来帮你分析。

等待用户回答。

### 2. 只读调查

AI 进入**纯只读模式**，禁止修改任何代码。

根据用户描述，执行以下调查（按需选择）：

| 调查类型 | 做什么 | 产出 |
|----------|--------|------|
| 代码结构分析 | 读 ARCHITECTURE.md + project-index.yaml（如存在）+ 相关代码文件 | 模块依赖图、分层问题 |
| 性能瓶颈定位 | 读日志/指标/慢查询 | 瓶颈清单 + 数据支撑 |
| 影响范围评估 | 读 DEPENDENCY_MAP + dependency-map.yaml（如存在）+ specs | 受影响的功能/文件列表 |
| 方案可行性对比 | 读现有代码 + 技术文档 | 方案对比表（优劣/成本/风险） |
| 规格差距分析 | 读 specs + 代码实现 | 规格与实现的不一致项 |

**优先使用索引**：如果 `project-index.yaml` 存在，先读取它快速定位相关文件，避免盲目扫描。

### 3. 输出探索报告

将探索报告写入 `changes/_explorations/{{YYYY-MM-DD}}-{{topic}}.md`，格式如下：

```markdown
## 探索报告

### 问题摘要
{{一句话概括发现的核心问题}}

### 调查结论
{{列出关键发现，每条附数据/代码引用}}

### 方案建议（如有）
| 方案 | 优点 | 缺点 | 影响范围 | 工作量估算 |
|------|------|------|---------|-----------|
| 方案 A | ... | ... | ... | 小 |
| 方案 B | ... | ... | ... | 中 |

### 建议的下一步
- [ ] 若选方案 A：执行 /harness-change（standard schema）
- [ ] 若选方案 B：执行 /harness-change（feature schema）
- [ ] 若需继续探索：描述下一个方向
```

### 4. 用户决策

用户可以选择：

1. **建立变更**：执行 `/harness-change`，将探索结论写入 proposal.md
2. **继续探索**：描述下一个方向，AI 继续调查
3. **放弃**：不做任何改动，探索报告仍保存到 `changes/_explorations/{{YYYY-MM-DD}}-{{topic}}.md`，供日后参考

若用户选择建立变更，AI 自动将探索报告的关键结论带入 proposal.md 的「动机」和「影响范围」章节。

## 约束

- **纯只读**：禁止修改任何代码文件
- **不生成变更文件夹**：探索结论不自动写入 changes/（报告存到 changes/_explorations/）
- **不更新 progress.md**：探索不算正式工作
- 探索过程中发现的其他问题，记录在报告中但不展开调查
