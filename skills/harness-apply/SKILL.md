# harness-apply：实现变更任务

## 触发条件

harness-change 完成后，用户要求实现变更时执行。

## 输入

- 变更名称（可选，可推断）
- `changes/{{change_name}}/tasks.md`（任务清单）
- 变更文件夹中的其他 artifacts（proposal, specs, design）

## 阶段 0：选择变更

如果未指定变更名称：

- 查看 `changes/` 目录下的活跃变更
- 只有一个 → 自动选择
- 多个 → 列出选项，让用户选择
- 无活跃变更 → 报告并停止

宣布：使用变更：`{{change_name}}`

---

## 阶段 1：只读分析（禁止写任何文件）

**目的**：在动代码前，先建立对"现状"和"差异"的理解，避免越界修改。

### 1.1 读取变更上下文

按以下顺序读取变更文件夹中的所有 artifacts：

1. `proposal.md`（理解动机和范围）
2. `delta-spec.md`（理解行为契约，如果存在）
3. `design.md`（理解技术方案，如果存在）
4. `tasks.md`（获取任务清单）

### 1.2 读取相关代码

针对 `tasks.md` 中列出的目标文件：

- 读取当前代码（**只读**，禁止任何修改）
- 记录与 proposal/spec 描述的差异
- 识别改动点周围的依赖文件（只读路径，不读取实现）

### 1.3 输出差异清单

```
## 阶段 1：只读分析报告

**变更**：{{change_name}}
**任务数**：{{total}}

### 任务清单
| # | 任务 | 目标文件 | 关联规格 | 关联代码现状 |
|---|------|---------|---------|------------|
| 1 | {{task}} | {{file}} | {{requirement}} | 现状一句话 |

### 差异/风险点
- {{risk_1}}（如：当前实现与 spec 假设不一致）
- {{risk_2}}（如：目标文件被多个任务共同依赖）

### 影响面
- 涉及模块：{{modules}}
- 涉及接口：{{apis}}
- 是否需要更新文档：{{docs_to_update}}
```

**暂停条件**：

- 任务清单有歧义 → 暂停，向用户确认
- 目标文件不存在 → 暂停，报告
- 差异超出 proposal 范围 → 暂停，建议新建变更

完成后**必须等待用户确认**才进入阶段 2（除非用户明确说"无需确认直接做"）。

---

## 阶段 2：写码（仅修改 tasks.md 指定文件）

**约束**：本阶段只触碰 `tasks.md` 中明确列出的文件，不顺手"优化"未列出的代码。

### 2.1 显示当前进度

```
## 实现中：{{change_name}}

阶段 2：写码
进度：{{completed}} / {{total}} 任务完成

待实现：
- [ ] {{task_1}}
- [ ] {{task_2}}
...
```

### 2.2 逐个实现任务

对每个未完成的任务：

1. 宣布正在处理的任务
2. **核对**：再次确认目标文件与阶段 1 差异清单一致（不一致 → 回到阶段 1）
3. 实现变更（创建/修改文件）
4. 保持变更最小化和聚焦
5. 在 `tasks.md` 中标记完成：`- [ ]` → `- [x]`
6. 继续下一个任务

**暂停条件**：

- 实现中发现设计问题 → 暂停，建议更新 artifacts（回到 harness-change）
- 任务涉及未在阶段 1 识别的文件 → 暂停，报告
- 遇到错误或阻塞 → 暂停，报告并等待

### 2.3 记录代码产出（写入 session-handoff.md）

全部任务完成后、进入阶段 3 前，立即更新 `session-handoff.md`：

```
## 阶段 2 产出（待自审）

变更：{{change_name}}
阶段 2 完成时间：{{timestamp}}

| 文件路径 | 操作 | 说明 |
|----------|------|------|
| {{file_1}} | 新建/修改 | {{task_1}} |

当前状态：阶段 2 完成，待执行阶段 3 自审。
恢复指令：从阶段 3 继续执行变更 {{change_name}}。
```

> 目的：若 apply 中途崩溃或会话断开，新会话可通过 session-handoff.md 恢复到阶段 3。

---

## 阶段 3：自审验证（对照 spec 检查实现）

**目的**：在标记实现完成前，自己检查一遍是否真符合规格。

### 3.1 规格对照（仅当 delta-spec.md 存在时，hotfix 跳过此步直接做 3.2）

针对 `delta-spec.md` 中每个需求：

- 在实现代码中找到对应位置
- 逐场景（前置/当/则）确认实现已覆盖
- 不一致 → 回到阶段 2 补完

### 3.2 影响面自检

对照阶段 1 的"影响面"清单：

- 涉及的接口是否已更新
- 涉及的文档是否已同步
- 未涉及的文件是否未被误改（可通过 git diff 核对）

### 3.3 输出自审报告 + 完成

```
## 自审报告

**变更**：{{change_name}}
**对照基准**：delta-spec.md + 阶段 1 影响面清单

### 规格覆盖
| 需求 | 场景数 | 已覆盖 | 备注 |
|------|-------|-------|------|
| {{req_1}} | {{n}} | ✓/✗ | |

### 影响面核对
- [ ] 接口已更新：{{api_list}}
- [ ] 文档已同步：{{doc_list}}
- [ ] 未误改范围外文件

### 结论
- ✓ 全部通过 → 进入完成输出
- ✗ 有遗漏 → 回到阶段 2 补完

## 实现完成

变更：{{change_name}}
进度：{{total}} / {{total}} 任务完成 ✓

本次完成：
- [x] {{task_1}}
- [x] {{task_2}}
...

下一步：执行 harness-verify 验证实现
```

## 约束

- **阶段 1 严格只读**：禁止修改任何文件
- **阶段 2 范围最小化**：只改 tasks.md 列出的文件
- **阶段 3 不修改代码**：只读对照，发现问题回阶段 2
- 每完成一个任务立即更新 `tasks.md` 的 checkbox
- 遇到阻塞不猜测，暂停并报告
- 三个阶段必须按顺序执行，不允许跳步
