# harness-execute：首次工单执行

## 触发条件

harness-order 完成后（或 harness-analyze 审计通过后），用户要求执行某个工单时使用。
也可在任意时刻执行指定工单。

## 输入

- 工单 ID（如 F01）或工单文件路径
- `orders/` 目录（工单文件）
- `AGENTS.md`（会话协议）

## 步骤

### 0. 选择 Agent（如适用）

如果当前会话支持 Agent 模式（多 Agent 协作），根据工单类型选择对应的 Agent：

| 工单类型 | 调用 Agent | 说明 |
|----------|-----------|------|
| 后端开发任务 | `backend-dev` | 后端代码实现 |
| 前端开发任务 | `frontend-dev` | 前端代码实现 |
| 混合任务 | 编排器自行处理 | 按文件类型分配给对应 Agent |

Agent 调用方式：将任务描述、文件清单、接口契约传递给对应 Agent，Agent 完成后返回结构化结果。编排器汇总结果并更新项目状态。

### 1. 选择工单

如果未指定工单：

- 读取 `progress.md` 找到下一个待执行工单
- 检查依赖是否全部完成
- 有未完成依赖 → 报告并建议先完成依赖工单

### 2. 读取工单文件

读取 `orders/{{id}}_{{title}}.md`，解析：

- 元数据（依赖、验证命令）
- 功能定义（行为、验证、状态流转）
- 执行指令（阶段 1/2/3 的详细步骤）

### 3. 执行阶段 1（只读）

严格按工单中的阶段 1 指令执行：

1. 读取列出的前序代码文件
2. 读取列出的文档
3. 输出差异清单和文件清单
4. 确认不做范围（不扩大、不缩小）

**阶段 1 完成后暂停**，向用户展示：

```
## 阶段 1 完成

差异清单：
- {{diff_item_1}}
- {{diff_item_2}}

计划实现的文件：
- {{file_1}}（新建/修改）
- {{file_2}}（新建/修改）

确认进入阶段 2？
```

### 4. 执行阶段 2（写码）

用户确认后，严格按工单中的阶段 2 指令执行：

1. 仅实现文件清单中的路径
2. 遵循 ARCHITECTURE.md 约束
3. 每完成一个功能点运行验证命令
4. 验证不通过 → 修复后重新验证

**阶段 2 完成后立即记录代码产出**：

更新 `session-handoff.md`，写入以下信息：

```
## 阶段 2 产出（待自审）

工单：{{id}} - {{title}}
阶段 2 完成时间：{{timestamp}}

| 文件路径 | 操作 | 说明 |
|----------|------|------|
| src/auth/login.py | 新建 | 登录接口 |
| src/models/user.py | 新建 | 用户模型 |

当前状态：阶段 2 完成，待执行阶段 3 自审。
恢复指令：从阶段 3 继续执行工单 {{id}}。
```

### 5. 执行阶段 3（自审）

写码完成后，按工单中的阶段 3 指令自审：

**逐项检查**：

- 完整性：所有行为已实现？验证通过？无 TODO？
- 正确性：匹配行为描述？边界覆盖？状态流转正确？
- 一致性：遵循架构？命名一致？目录符合约定？

**自审不通过 → 修复 → 重新自审（最多 3 轮）**

### 6. 更新项目状态

自审通过后：

1. 更新 `progress.md`：标记工单已完成
2. 更新 `session-handoff.md`：将阶段 2 的临时记录替换为最终状态（标记已验证，添加自审结果）
3. 更新 `feature_list.json`：标记功能完成
4. 回填文档中的 TBD 占位符（如有）

**可选：执行 harness-project-memory**

如果用户要求积累项目经验（或工单涉及架构决策、新约定、技术选型变更）：
- 提取本次工单的决策、约定、教训
- 更新 `docs/meta/memory/` 下的记忆文件
- 此步骤不阻塞后续流程

### 7. 输出完成报告

```
## 工单完成

工单：{{id}} - {{title}}
状态：已完成
验证命令：{{verify_command}} ✓

自审结果：
- 完整性：✓
- 正确性：✓
- 一致性：✓

TBD 回填：{{count}} 个
下一个工单：{{next_order_id}}
```

**全部工单完成时**（无下一个工单），额外输出：

```
🎉 全部 {{total}} 个工单已完成！

建议执行集成验证：
1. 运行全量测试：{{verify_commands.health_check}}
2. 检查架构合规：{{verify_commands.architecture_check}}（如配置了的话）
3. 启动服务确认端到端流程正常

验证通过后，后续代码变更请使用 Flow B：
/harness-change → /harness-apply → /harness-verify → /harness-archive
```

## 约束

- 严格按工单的三段式执行，不跳阶段
- 阶段 1 禁止写代码
- 阶段 2 仅实现清单内文件
- 阶段 3 自审不通过必须修复（最多 3 轮）
- 不修改其他工单的内容
